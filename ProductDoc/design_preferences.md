# Pobi APP 设计偏好

> 文档用途：作为 Pobi 后续视觉探索、Figma 设计稿、交互原型和 iOS 实现的统一设计参考。  
> 适用平台：iOS，优先使用 UIKit / SwiftUI 原生能力。  
> 当前版本：V1.0  
> 更新日期：2026-07-21

## 1. 设计定位

Pobi 的整体视觉应当简约、清晰、内容优先，并在关键局部体现鲜明设计感。整体气质参考 Apple 官方应用的设计方式，例如 App Store：基础框架克制、系统感强，关键内容、主入口和主题区域具有足够的视觉识别度。

设计目标不是让每个区域都“有设计”，而是让用户在熟悉的 iOS 使用体验中，通过少量高质量的品牌元素感知 Pobi 的独特性。

核心关键词：

- 简约
- 原生
- 内容优先
- 高饱和强调
- 局部精致
- 清晰层级
- 轻量动效
- 品牌化关键入口

## 2. 核心设计原则

### 2.1 整体简约，局部突出

- 页面基础框架保持简洁，以内容、留白、排版和层级组织界面。
- 不为普通信息增加多余卡片、描边、渐变、阴影或装饰。
- 每个页面只设置一个主要视觉焦点，通常是当前任务、核心内容或主要操作。
- 设计表现集中在主按钮、关键入口、空状态、阶段状态和重要内容展示区域。
- 避免所有模块使用相同重量的卡片样式，防止界面平均用力。

### 2.2 系统体验优先

- 页面框架、导航、弹窗、菜单、列表、表单、选择器和动画优先使用 UIKit / SwiftUI 系统样式与交互习惯。
- 用户应当无需重新学习基本交互方式。
- 自定义设计不得破坏系统手势、安全区域、动态字体、深色模式和辅助功能。
- 只有在系统组件无法表达 Pobi 的品牌特征或核心任务时，才增加自定义视觉层。

### 2.3 内容决定组件形式

- 根据内容语义合理选择纯文本、图标、`icon + text` Label 或图文组合。
- 图标用于提高识别速度，文本用于消除歧义；两者不应机械地同时出现。
- 不用装饰性图标填满界面，也不让用户依赖抽象图标猜测关键功能。
- 数字、单词、学习状态和任务标题应通过排版形成主次，而不是全部放入独立容器。

### 2.4 品牌设计集中在关键视图

- 主操作按钮、核心入口、空状态、学习阶段状态和重要占位视图必须体现 Pobi 的独特设计语言。
- 品牌识别优先通过固定的强调色、标志性图标、字体层级、形状或动效建立。
- 同一种品牌元素应跨页面重复使用，形成稳定认知，不为每个页面创造不同视觉语言。

## 3. 视觉风格

### 3.1 页面基础

- 使用 iOS 系统背景色、分组背景、材质和分隔方式建立基础页面层级。
- 以大面积中性色或系统背景承载内容，高饱和色用于关键局部。
- 优先通过留白、对齐、字号、字重和内容分组划分区域。
- 分隔线用于列表或连续内容；独立卡片只用于真正独立的内容对象。
- 阴影应轻微且少量使用，主要用于浮层、悬浮操作或确有层级关系的视图。
- 圆角应参考系统组件比例，同类组件保持统一，避免过度圆润或所有内容胶囊化。

### 3.2 设计感来源

Pobi 的设计感主要来自以下组合，而不是复杂装饰：

1. 清晰且具有节奏的排版。
2. 高饱和品牌强调色与中性背景的对比。
3. 统一的 SF Symbols 图标语言。
4. 关键入口中的品牌图形或特殊状态表达。
5. 自然、克制的系统动效。
6. 真实学习内容带来的视觉变化。

## 4. 色彩偏好

### 4.1 色彩策略

- 使用一套高饱和度主品牌色，承担主要按钮、选中状态、关键进度和重要入口。
- 可设置一套辅助高饱和色，用于区分学习、复习或其它明确语义，但单个页面不应出现过多竞争色。
- 高饱和色优先小面积、高价值使用，不建议铺满整页背景或普通信息卡。
- 普通文本、背景、分隔线和次要状态优先使用 iOS 语义色，而不是固定灰度值。
- 成功、警告、错误等状态使用系统语义色，不随意挪用品牌色替代。

### 4.2 色彩层级

- 主品牌色：核心 CTA、当前阶段、选中态、关键进度。
- 辅助品牌色：次级主题、辅助强调、插画或数据区分。
- 系统中性色：页面背景、列表、普通卡片、正文和分隔线。
- 系统语义色：成功、提醒、错误、禁用和权限状态。

### 4.3 色彩约束

- 同一页面原则上只允许一个最强强调色。
- 不使用多组高饱和渐变同时竞争注意力。
- 不以颜色作为表达状态或操作主次的唯一方式；同时使用文字、位置、形状或图标。
- 所有颜色需要分别验证浅色模式、深色模式及无障碍对比度。
- 具体色值在高保真视觉方向确认后定义，本偏好文档不预设 HEX 值。

## 5. 排版偏好

- 默认使用 iOS 系统字体：英文和数字使用 SF Pro，中文使用系统中文字体。
- 优先使用系统字号体系、动态字体和语义化文本样式。
- 通过字号、字重、颜色和间距建立层级，不依赖大量不同字体。
- 大标题和关键数字可使用更强字重与更紧凑的字距，普通正文保持舒适阅读。
- 主任务标题、单词本身、学习数字和阶段状态允许形成具有品牌感的特殊排版组合。
- 若未来引入品牌字体，仅用于 Logo、品牌标题或极少量展示文字，不替换正文系统字体。
- 文案变长或系统字体放大时，组件应优先增高或换行，不截断关键操作与状态信息。

## 6. Icon 与 Label

### 6.1 图标来源

- 默认使用 Apple SF Symbols。
- SF Symbols 无法表达的品牌或业务图标，使用相同视觉风格的自定义图标。
- 自定义图标需匹配 SF Symbols 的线宽、圆角、视觉重心、网格和渲染方式。
- 同一页面中的图标应统一使用相近的 weight 与 rendering mode。

### 6.2 渲染方式

- 普通工具与导航图标优先使用 Monochrome 或 Hierarchical。
- 关键状态或品牌入口可使用 Palette 或有限的 Multicolor，但必须保持克制。
- 图标颜色优先使用语义前景色、tint 或品牌色，不使用无意义的彩色图标集合。

### 6.3 Icon、Label 与 Text 的选择

- 仅使用 icon：用户高度熟悉、语义明确且空间有限的操作，例如关闭、返回、播放。
- 使用 `icon + text` Label：导航入口、设置项、主要功能入口和需要快速识别的操作。
- 仅使用 text：标题、说明、数据、状态和语义已足够清楚的操作。
- 关键操作不能仅使用难以理解的自定义图标；必须同时提供文字或辅助标签。
- 图标与文本应作为一个整体对齐，不让图标成为独立装饰。

## 7. 系统组件偏好

### 7.1 导航

- 优先使用系统 NavigationBar、NavigationStack 和标准返回手势。
- 标题根据内容层级使用 inline 或 large title，不自定义不必要的标题栏高度。
- 页面主操作不应全部堆叠在 NavigationBar 中。
- 菜单优先使用系统 Menu、Context Menu 或 Popover；确需品牌展示时再增加轻量自定义。

### 7.2 列表与表单

- 设置、Profile、选择器等信息密集页面优先采用系统 List / Form 的分组和行高逻辑。
- 行项目优先使用系统 Label 结构：前置图标、标题、可选副标题、尾部状态或 Chevron。
- 开关、日期、时间、选择器优先使用系统控件。
- 不使用网页式的大量描边输入框替代原生表单体验。

### 7.3 弹窗与覆盖层

- 确认和警示优先使用 Alert / Confirmation Dialog。
- 需要持续浏览或编辑的内容优先使用 Sheet，并合理使用系统 Detent。
- 轻量选择和快捷操作优先使用 Menu / Popover。
- 全屏覆盖仅用于沉浸式学习、完整流程或不可被小型弹窗承载的任务。
- 弹窗不自定义复杂框架，品牌化应集中在图标、标题、关键插图或主要操作上。

### 7.4 按钮

- 主要按钮使用高饱和品牌色、清晰动词文案和足够大的点击区域。
- 次要按钮使用系统 bordered、tinted、plain 等层级，避免与主按钮竞争。
- 破坏性操作严格使用系统 destructive 语义样式。
- 同一页面只保留一个视觉权重最高的主要按钮。

## 8. 关键视图的品牌化

以下视图必须比普通系统界面具有更明确的 Pobi 识别度：

- 首页当前学习任务。
- 开始学习、开始复习等主要按钮。
- 学习计划创建入口。
- 无计划、无内容等空状态。
- 学习阶段切换和会话完成反馈。
- 单词学习与记忆卡片。
- 学习进度与关键统计入口。

### 8.1 可使用的品牌元素

- 高饱和 Pobi 主色。
- Pobi 标志性图标或图形母题。
- 特定的标题/数字排版组合。
- 具有一致规则的背景色块、圆角或局部材质。
- 与学习、记忆相关的轻量插画或符号。
- 固定的进入、完成和状态切换动效。

### 8.2 使用规则

- 每个关键视图选择 1–2 种品牌元素即可，不同时堆叠颜色、渐变、插画、纹理和复杂动效。
- 同类关键视图应复用同一设计规则，例如所有主要学习入口使用一致的形状和动效。
- 品牌元素不能降低文字可读性或改变用户对系统控件的预期。
- Pobi 的标志性图形、主色和关键动效需在高保真视觉方向阶段确定。

## 9. 动画与转场偏好

- 优先采用 UIKit / SwiftUI 系统转场、Sheet 动画、导航动画和控件反馈。
- 自定义动画使用自然的 spring、easeInOut 和 matched geometry 逻辑，保持短促、连续、可预测。
- 动画用于解释状态变化、强化操作结果或保持空间关系，不作为装饰循环播放。
- 关键学习入口、卡片切换、完成反馈可具有 Pobi 独特动效，但不得拖慢学习节奏。
- 点击后应立即提供按压、颜色、缩放或材质反馈。
- 支持系统“减少动态效果”，必要时以淡入淡出替代位移、缩放和弹性动画。

## 10. 页面信息组织

- 页面优先展示用户当前最需要理解或执行的内容。
- 单个页面只设置一个主任务或主 CTA。
- 相关信息就近分组，不将所有信息都包装为独立卡片。
- 主要内容区可以合理组合：
  - 标题 + 说明。
  - SF Symbol + Label。
  - 数字 + 单位 + 状态。
  - 图标 + 标题 + 副标题 + Chevron。
  - 进度 + 当前任务 + 下一步行动。
- 次要信息应通过字重、颜色和位置弱化，不仅通过缩小字号。
- 真实内容优先于装饰元素；在内容不足时才使用插画或品牌占位。

## 11. 可访问性与平台适配

- 支持 Dynamic Type，主要流程在较大字号下仍可完成。
- 所有点击区域至少 44 × 44 pt。
- 正文与背景对比度至少满足 WCAG AA。
- 图标按钮必须提供 VoiceOver 标签。
- 颜色不是唯一状态提示。
- 支持浅色和深色模式；必要时使用不同的品牌色变体保证观感与对比度。
- 尊重“减少动态效果”“增强对比度”“降低透明度”等系统辅助设置。
- 页面适配 iPhone 不同宽度、安全区域和横竖屏策略；V1 以竖屏手机为主要设计目标。

## 12. 应避免的设计

- 避免所有模块都使用大圆角卡片和阴影。
- 避免同时使用多组高饱和颜色争夺注意力。
- 避免大面积渐变、玻璃拟态和纯装饰性背景。
- 避免自定义系统已有且用户熟悉的开关、菜单、弹窗和选择器。
- 避免使用 Emoji、非统一图标库或不同线宽的图标混搭。
- 避免只有图标、没有文字且语义不清的关键入口。
- 避免为了“设计感”牺牲信息密度、可读性和操作效率。
- 避免把 Android、Web 或游戏 UI 语言直接移植到 iOS。
- 避免每个页面采用不同的品牌色、插画风格或圆角规则。
- 避免通过复杂动效打断连续学习流程。

## 13. 后续视觉探索要求

高保真视觉方向探索应以本文件为共同约束，并在以下方面形成明显差异：

- Pobi 主品牌色与辅助色组合。
- 关键任务入口的品牌图形与版式。
- 学习/复习状态的视觉区分方式。
- 进度与统计的内容表达形式。
- 空状态和学习卡片的插画或符号风格。

视觉方案可以有不同气质，但都必须满足：

- 基础框架具有 Apple 官方应用的原生感。
- 主要组件能够合理映射至 UIKit / SwiftUI。
- 高饱和色集中用于关键局部。
- SF Symbols 是主要图标语言。
- 首页存在明确的单一主任务。
- 关键入口具备可跨页面复用的 Pobi 品牌识别。

## 14. 设计评审清单

### 整体

- [ ] 页面是否整体简约，并只有少量设计焦点？
- [ ] 是否具有清晰的 iOS 原生应用观感？
- [ ] 是否优先使用系统组件与交互模式？
- [ ] 是否避免卡片、圆角、阴影和颜色的过度使用？

### 内容与层级

- [ ] 用户能否立即识别当前页面的主任务？
- [ ] Icon、Label 和 Text 的组合是否合理？
- [ ] 关键信息是否通过排版而非容器堆叠建立层级？
- [ ] 次要操作是否明显弱于主要操作？

### 品牌与视觉

- [ ] 高饱和色是否集中在关键视图？
- [ ] 主按钮、重要入口和占位视图是否具有 Pobi 识别度？
- [ ] 品牌元素是否跨页面复用并保持一致？
- [ ] SF Symbols 与自定义图标是否属于同一视觉语言？

### 平台与可访问性

- [ ] 弹窗、菜单、动画和选择器是否符合 iOS 使用习惯？
- [ ] 是否支持动态字体、深色模式和 VoiceOver？
- [ ] 点击区域和颜色对比是否符合可访问性要求？
- [ ] 减少动态效果后是否仍能正确理解状态变化？
